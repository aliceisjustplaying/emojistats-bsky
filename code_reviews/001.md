## 🔴 CRITICAL ISSUES

### 1. Rate Limiter Global Bucket Misconfiguration

**Location**: `packages/backfill/backfill/util/xrpc.ts:63`

```typescript
const GLOBAL_LIMITER = new RateLimiter(3000, 10); // 3000 requests / 5 minutes
```

**Issue**: The refill rate is incorrectly configured. The code claims "3000 requests / 5 minutes" but with a refill rate of 10 req/sec, the actual capacity over 5 minutes would be:

- 10 req/sec × 300 seconds = 3,000 tokens refilled
- Plus initial 3,000 token capacity = 6,000 total requests per 5 minutes

However, if the intent is to limit to exactly 3,000 requests per 5 minutes, the refill rate should be:

- 3,000 / 300 = 10 req/sec ✓ (Actually correct!)

**Clarification Needed**: The math is correct, but the comment is misleading. The limiter allows 3,000 **initial** tokens plus 10/sec refill, giving ~6,000 requests per 5 minutes after warmup.

**Impact**: May exceed intended API rate limits from Bluesky, potentially causing account throttling or bans.

**Recommendation**:

```typescript
// Option 1: True 3000 req/5min limit
const GLOBAL_LIMITER = new RateLimiter(3000, 10); // 3000 initial + 3000 over 5min = 6000/5min

// Option 2: Strict 3000 req/5min limit
const GLOBAL_LIMITER = new RateLimiter(0, 10); // 0 initial + 3000 over 5min = 3000/5min

// Option 3: Match documented behavior with correct comment
const GLOBAL_LIMITER = new RateLimiter(3000, 10); // ~6000 requests per 5 minutes (3k burst + 10/sec)
```

### 2. RateLimiter Race Condition

**Location**: `packages/backfill/backfill/runner.ts:278-309` and `packages/backfill/backfill/util/xrpc.ts:30-61`

**Issue**: The `RateLimiter` class has a race condition in the `take()` method:

```typescript
async take() {
  this.refill();
  while (this.tokens < 1) {
    await new Promise((resolve) => setTimeout(resolve, 50));
    this.refill();
  }
  this.tokens -= 1;  // ⚠️ NOT ATOMIC
}
```

Between the `while` check and `this.tokens -= 1`, another concurrent caller could also pass the check and decrement, potentially making `tokens` negative. While JavaScript is single-threaded, the `await` yields control, allowing interleaving.

**Impact**: Multiple concurrent requests could "borrow" tokens, exceeding rate limits. In high-concurrency scenarios (default 64 concurrent repos), this could lead to significant overage.

**Proof**: With 64 concurrent workers, if 10 workers simultaneously pass the `while` check when `tokens = 1`, all 10 will decrement, resulting in `tokens = -9`.

**Recommendation**:

```typescript
class RateLimiter {
  private tokens: number;
  private lastRefill: number;
  private pending: Array<() => void> = [];
  private processing = false;

  async take() {
    return new Promise<void>((resolve) => {
      this.pending.push(resolve);
      this.processPending();
    });
  }

  private async processPending() {
    if (this.processing) return;
    this.processing = true;

    while (this.pending.length > 0) {
      this.refill();
      if (this.tokens >= 1) {
        this.tokens -= 1;
        const resolve = this.pending.shift()!;
        resolve();
      } else {
        await new Promise((resolve) => setTimeout(resolve, 50));
      }
    }

    this.processing = false;
  }

  private refill() {
    const now = Date.now();
    const elapsed = (now - this.lastRefill) / 1000;
    if (elapsed <= 0) return;
    this.tokens = Math.min(
      this.capacity,
      this.tokens + elapsed * this.refillPerSec,
    );
    this.lastRefill = now;
  }
}
```

### 3. Unbounded Memory Growth in limiterByPds Map

**Location**: `packages/backfill/backfill/runner.ts:46,231-237`

**Issue**: The `limiterByPds` map grows indefinitely as new PDS hosts are discovered:

```typescript
private readonly limiterByPds = new Map<string, RateLimiter>();

private getLimiter(pds: string) {
  let limiter = this.limiterByPds.get(pds);
  if (!limiter) {
    limiter = new RateLimiter(20, 20);
    this.limiterByPds.set(pds, limiter);  // ⚠️ Never cleaned up
  }
  return limiter;
}
```

**Impact**: With thousands of PDS hosts, each maintaining a `RateLimiter` instance (with timers and state), memory usage will grow continuously. Over a full backfill of Bluesky's network (~2,700+ PDS hosts currently), this could consume significant memory.

**Calculation**:

- 2,700 PDS hosts
- ~200 bytes per RateLimiter instance
- ~540 KB total (not huge, but grows indefinitely)

**Recommendation**: Implement LRU eviction:

```typescript
import { LRUCache } from 'lru-cache';

private readonly limiterByPds = new LRUCache<string, RateLimiter>({
  max: 500, // Keep only recently used limiters
  ttl: 1000 * 60 * 30, // 30 minutes
});

private getLimiter(pds: string) {
  let limiter = this.limiterByPds.get(pds);
  if (!limiter) {
    limiter = new RateLimiter(20, 20);
    this.limiterByPds.set(pds, limiter);
  }
  return limiter;
}
```

### 4. Database Connection Pool Exhaustion Risk

**Location**: `packages/backfill/backfill/db.ts:12-19`

**Issue**: No connection pool limits configured:

```typescript
export function createPool({ databaseUrl, schema }: DatabaseDeps) {
  const pool = new Pool({ connectionString: databaseUrl });
  // ⚠️ No max connections limit, timeout, or error handling
  const searchPath = quoteIdent(schema);
  pool.on("connect", (client) => {
    void client.query(`SET search_path TO ${searchPath}`);
  });
  return pool;
}
```

With default concurrency of 64 repos, and each repo potentially holding a DB connection during `insertEmojiRows()` + dimension lookups, you could easily exhaust the default PostgreSQL connection limit (100).

**Impact**: Connection exhaustion leading to hangs, timeouts, and cascading failures.

**Recommendation**:

```typescript
export function createPool({ databaseUrl, schema }: DatabaseDeps) {
  const pool = new Pool({
    connectionString: databaseUrl,
    max: 20, // Limit total connections
    idleTimeoutMillis: 30000, // Close idle connections after 30s
    connectionTimeoutMillis: 5000, // Fail fast if no connection available
  });

  pool.on("error", (err) => {
    console.error("Unexpected database pool error:", err);
  });

  const searchPath = quoteIdent(schema);
  pool.on("connect", async (client) => {
    try {
      await client.query(`SET search_path TO ${searchPath}`);
    } catch (err) {
      console.error("Failed to set search_path:", err);
    }
  });
  return pool;
}
```

### 5. fetchRepoStreamWithBackoff Retry Logic Flaw

**Location**: `packages/backfill/backfill/runner.ts:240-267`

**Issue**: The backoff loop continues even after deadline expires, and may retry non-idempotent operations:

```typescript
private async fetchRepoStreamWithBackoff(
  descriptor: RepoDescriptor,
  deadline: number,
) {
  let lastError: unknown;
  for (const delaySeconds of NETWORK_BACKOFF_SECONDS) {
    if (Date.now() > deadline) {
      break;
    }
    if (delaySeconds > 0) {
      await new Promise((resolve) =>
        setTimeout(resolve, delaySeconds * 1000),
      );
      if (Date.now() > deadline) {  // ⚠️ Only checks AFTER sleeping
        break;
      }
    }
    try {
      return await this.fetchRepoStream(descriptor);
    } catch (error) {
      if (!isConnectionError(error)) {
        throw error;
      }
      lastError = error;
    }
  }
  throw lastError ?? new RepoTimeoutError(descriptor.did);
}
```

**Problem**: If deadline expires during the `setTimeout`, we still check deadline AFTER waiting, potentially exceeding the timeout significantly (up to 128 extra seconds on the last retry).

**Impact**: Repos can hang for much longer than `repoProcessingTimeoutMs` (default 5 minutes), up to 5 minutes + 128 seconds = ~7.1 minutes.

**Recommendation**:

```typescript
private async fetchRepoStreamWithBackoff(
  descriptor: RepoDescriptor,
  deadline: number,
) {
  let lastError: unknown;
  for (const delaySeconds of NETWORK_BACKOFF_SECONDS) {
    const now = Date.now();
    if (now >= deadline) {
      break;
    }

    if (delaySeconds > 0) {
      const maxWait = deadline - now;
      const actualWait = Math.min(delaySeconds * 1000, maxWait);
      if (actualWait <= 0) break;
      await new Promise((resolve) => setTimeout(resolve, actualWait));
    }

    try {
      return await this.fetchRepoStream(descriptor);
    } catch (error) {
      if (!isConnectionError(error)) {
        throw error;
      }
      lastError = error;
    }
  }
  throw lastError ?? new RepoTimeoutError(descriptor.did);
}
```

### 7. Batch Flush Failure Handling

**Location**: `packages/backfill/backfill/writer.ts:46-51`

**Issue**: If `insertEmojiRows()` fails, the batch is lost:

```typescript
async flush() {
```

```typescript
async flush() {
```

### 8. CAR Stream Error Detection Too Narrow

**Location**: `packages/backfill/backfill/runner.ts:357-366`

**Issue**: Only catches specific varint/block errors:

```typescript
function isCarStreamError(error: unknown): boolean {
  if (!error || typeof error !== "object") return false;
  const message = (error as any).message ?? "";
  if (typeof message !== "string") return false;
  return (
    /unexpected eof while decoding varint/i.test(message) ||
    /invalid varint/i.test(message) ||
    /invalid block/i.test(message)
  );
}
```

**Missing Patterns**:

- "unexpected end of stream"
- "CID not found"
- "DAG-CBOR decode error"
- "invalid CAR header"

**Impact**: Other CAR corruption scenarios will be treated as unknown failures, incrementing `unknownFailures` metric instead of `transientFailures`, and repo won't be retried.

**Recommendation**:

```typescript
function isCarStreamError(error: unknown): boolean {
  if (!error || typeof error !== "object") return false;
  const message = ((error as any).message ?? "").toLowerCase();
  if (typeof message !== "string") return false;
  return (
    message.includes("varint") ||
    message.includes("invalid block") ||
    message.includes("unexpected eof") ||
    message.includes("unexpected end of stream") ||
    message.includes("car") ||
    message.includes("dag-cbor") ||
    message.includes("cid not found")
  );
}
```

### 12. Emoji Regex Catastrophic Backtracking Risk

**Location**: `packages/backfill/backfill/postExtractor.ts:6,24`

**Issue**: Using `emoji-regex` on untrusted user input:

```typescript
const emojiRegex = emojiRegexFactory();

// Later:
const emojiMatches = text.match(emojiRegex) ?? [];
```

While `emoji-regex` is generally safe, it's a complex regex. If fed specially crafted input, it _could_ exhibit catastrophic backtracking.

**Impact**: Potential DoS if malicious posts cause excessive regex processing time.

**Recommendation**: Add timeout wrapper:

```typescript
function safeMatchEmoji(text: string, timeoutMs = 1000): string[] | null {
  const start = Date.now();
  try {
    return text.match(emojiRegex) ?? [];
  } catch (error) {
    if (Date.now() - start > timeoutMs) {
      console.warn(`Emoji regex timeout on text of length ${text.length}`);
      return null;
    }
    throw error;
  }
}
```

Better: use a streaming emoji parser that doesn't rely on regex.

### 13. Parquet File Naming Collision Risk

**Location**: `packages/backfill/backfill/parquetSink.ts` (implied by git diff)

**Issue**: Parquet files use timestamp-based naming:

```
emoji_posts_2025-11-10T17-44-16-535Z.parquet
```

If two backfill processes start within the same millisecond (unlikely but possible in automated deployments), they'll overwrite each other.

**Recommendation**: Add process ID or random suffix:

```typescript
const filename = `emoji_posts_${new Date().toISOString().replace(/[:.]/g, "-")}_${process.pid}_${Math.random().toString(36).slice(2, 8)}.parquet`;
```

## PERFORMANCE ANALYSIS

### Throughput Estimation

**Configuration**:

- Concurrency: 64 repos in parallel
- Rate limits:
  - Per-PDS: 20 req/sec (getRepo)
  - Global: ~6,000 req/5min (other atproto calls)

**Bottleneck Analysis**:

1. **Network I/O** (main bottleneck)

   - Each repo requires 1 × `getRepo` call
   - At 20 req/sec per PDS, with ~2,700 PDS hosts, theoretical max = 54,000 repos/sec
   - **Reality**: Most repos are on `bsky.social` PDS, so effective rate is ~20 repos/sec per PDS
   - If 80% of repos are on top 10 PDS hosts: ~200 repos/sec max

2. **Database Writes**

   - Batch size: 500 rows
   - Batch insert time: ~50-100ms (estimated)
   - Throughput: ~5,000-10,000 rows/sec
   - With 64 concurrent workers, this is NOT the bottleneck

3. **CAR Parsing**
   - Depends on repo size
   - Small repos: ~10ms
   - Large repos: ~500ms
   - **Average**: ~50ms per repo
   - Throughput with 64 workers: ~1,280 repos/sec
   - NOT the bottleneck

**Conclusion**: Network rate limits are the primary bottleneck. The system is well-designed for this constraint.

### Memory Consumption

**Estimated Memory Usage**:

- Each concurrent worker holds:
  - CAR stream: ~1-10 MB (depends on repo size)
  - Batch buffer: ~500 rows × 500 bytes = 250 KB
  - Misc overhead: ~100 KB
- Total per worker: ~1-10 MB
- With 64 workers: ~64-640 MB

**Database Connection Pool**:

- Default pool with no limit: **CRITICAL ISSUE** (see issue #4)
- With recommended limit of 20: ~20 × 10 KB = 200 KB

**Total Estimated**: 500 MB - 1 GB under normal load.

### Optimization Opportunities

1. **Parallelize Dimension Lookups** (`writer.ts:18-22`)

   ```typescript
   // Current (sequential):
   const langId = await this.dimensions.getLanguageId(post.primaryLang);
   const clientId = await this.dimensions.getClientId(post.clientIdentifier);
   const emojiIds = await Promise.all(
     post.emojiGlyphs.map((glyph) => this.dimensions.getEmojiId(glyph)),
   );

   // Optimized (parallel):
   const [langId, clientId, emojiIds] = await Promise.all([
     this.dimensions.getLanguageId(post.primaryLang),
     this.dimensions.getClientId(post.clientIdentifier),
     Promise.all(
       post.emojiGlyphs.map((glyph) => this.dimensions.getEmojiId(glyph)),
     ),
   ]);
   ```

   **Estimated Speedup**: 2-3× per post (if cache misses occur).

2. **Increase Batch Size** (if DB can handle it)

   - Current: 500
   - Try: 1,000 or 2,000
   - **Tradeoff**: Higher memory usage, longer flush delays

3. **Connection Pooling for XRPC Clients**
   - Currently, each PDS gets a separate `Client` instance, which is good.
   - But consider using HTTP/2 or connection pooling library for multiplexing.

### Handled Well ✅

2. **Corrupt CAR streams** - Caught and logged as transient (could be improved, see issue #8)

   - **Recommendation**: Add retry logic for `40P01` error code

2. **Postgres connection loss mid-transaction** - Pool will reconnect, but current batch lost
